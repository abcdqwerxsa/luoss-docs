# 开发环境

开发环境提供基于 SSH 的云端开发环境，支持使用本地 IDE（VS Code Remote-SSH、PyCharm 等）远程连接，进行代码编写、调试和模型训练。

<FeatureBadge status="stable" />

## 功能概述

| 功能 | 说明 |
|------|------|
| **创建环境** | 配置资源创建通用开发环境 |
| **启动/停止** | 控制环境运行状态 |
| **SSH 访问** | 通过 SSH 连接使用本地 IDE 开发 |
| **工作盘（数据盘）** | 环境的主目录持久化盘由[数据盘](/guide/data-volumes)统一管理，可新建或复用 |
| **全量持久化** | 容器内所有修改重启后保留（pip/apt install 均不丢失） |
| **存储扩容** | 工作盘支持在线扩容 |
| **搜索排序** | 按名称、状态快速查找和排序 |
| **NPU 访问** | 自动访问所有 NPU 设备 |

::: tip 篇幅限制
每个用户最多创建 **2 个**开发环境。如需创建新环境，请先删除不需要的环境。
:::

| 字段 | 说明 |

## 创建环境

### 步骤

1. 点击 **创建环境** 按钮
2. 填写基本信息：

| 字段 | 说明 |
|------|------|
| 镜像模板 | 选择开发环境使用的镜像模板（默认模板已预选） |
| 环境名称 | 平台当前仅允许创建指定名称的个人环境（如 `demo1`） |
| 工作盘 | 环境的主目录持久化盘，二选一（见下方） |

3. 选择**工作盘**（环境主目录的持久化存储，由[数据盘](/guide/data-volumes)统一管理）：
   - **新建数据盘**：填写容量（GiB）与名称，平台创建一块新数据盘作为工作盘。
   - **使用空闲数据盘**：从下拉列表中选择一块已有的空闲数据盘复用为工作盘——适合在环境间复用同一份数据/配置，或提前在「数据盘」页面准备好。
4. 点击 **创建**

::: info 通用开发环境
开发环境自动使用统一模板，具有以下特性：
- **全量持久化**：容器内所有修改（pip install、apt-get、系统配置等）在环境重启后均会保留，持久化空间即工作盘容量（创建时自选）
- 已启用特权模式和 Docker-in-Docker 支持
- 可访问所有 NPU 设备（无需手动指定数量）
- 预装常用开发工具（conda、Python 等）
:::

::: tip 工作盘即数据盘
工作盘本质上就是一块[数据盘](/guide/data-volumes)，因此：
- 可在「数据盘」页面统一查看、管理、复用，**不再产生删除环境后遗留的孤儿卷**。
- 删除环境时仅**解绑**工作盘，数据保留，可挂到下一个环境继续使用。
- 工作盘容量计入数据盘的 [2 TiB 配额](/guide/data-volumes#创建数据盘)。
:::

## SSH 连接

开发环境通过 SSH 提供访问，您可以使用本地 IDE 远程连接到云端环境。

::: tip 会话审计与录像
通过堡垒机的 SSH 连接会被平台**全程审计**：连接记录入审计日志，交互式终端会话会被录像（可回放），管理员可实时监控。仅交互式 shell（含 VS Code 集成终端）被录制，VS Code 主连接 / sftp / scp 不录制。详见 [SSH 会话审计](/admin/ssh-sessions)。
:::

### 前提条件

1. **上传 SSH 公钥**：在 [设置 > SSH 公钥](/guide/settings#ssh-公钥管理) 中添加您的公钥
2. **环境处于运行状态**：确保开发环境已启动（状态为 Running）

### 连接方式

在环境详情页面获取 SSH 连接命令：

```bash
ssh -p <端口> root@<服务器地址>
```

::: tip 连接信息
每个环境的 SSH 连接命令在环境详情页面中显示，您可以直接复制使用。
:::

### 使用 VS Code Remote-SSH 连接

1. 安装 VS Code 扩展：**Remote - SSH**（`ms-vscode-remote.remote-ssh`）
2. 按 `Ctrl+Shift+P` 打开命令面板，输入 `Remote-SSH: Open Configuration File...`
3. 添加 SSH 配置：

```
Host luoss-env
    HostName <服务器地址>
    Port <端口>
    User root
    IdentityFile ~/.ssh/id_ed25519
```

4. 按 `Ctrl+Shift+P`，选择 `Remote-SSH: Connect to Host...`
5. 选择 `luoss-env` 连接

### 使用 PyCharm 连接

1. 打开 PyCharm，进入 **Settings > Tools > SSH Configurations**
2. 添加新配置：
   - Host：服务器地址
   - Port：SSH 端口
   - User name：`root`
   - Authentication type：OpenSSH config and authentication agent
3. 测试连接并保存

### SCP / SFTP / rsync 文件传输

本地与开发环境之间可用 `scp`、`sftp`、`rsync` 传输文件，连接参数与 SSH 完全一致（同一堡垒机地址、端口、用户名）。

::: tip 端口标志的大小写
- `ssh` 用**小写** `-p` 指定端口；`scp` / `sftp` 用**大写** `-P`。
- `rsync` 的端口是交给 ssh 处理的，所以仍是小写 `-p`，写在 `-e "ssh -p <端口>"` 里。
:::

`@` 前的用户名即**环境名**（见 [多环境路由](#多环境路由)），`root` 表示默认环境：

```bash
# 上传单个文件
scp -P <端口> 本地文件.txt <环境名>@<服务器地址>:/models/

# 从环境下载文件
scp -P <端口> <环境名>@<服务器地址>:/models/output.txt ./

# 上传整个目录（-r 递归）
scp -P <端口> -r 本地目录/ <环境名>@<服务器地址>:/models/

# 交互式 SFTP（进入后用 ls / put / get）
sftp -P <端口> <环境名>@<服务器地址>

# rsync：增量、断点续传、带进度，同步整棵目录树首选
rsync -av --progress -e "ssh -p <端口>" 本地目录/ <环境名>@<服务器地址>:/models/
```

::: tip rsync 末尾斜杠
`本地目录/`（带斜杠）表示同步目录**内容**到目标；`本地目录`（不带斜杠）表示在目标下新建一个同名子目录。与 `cp` 语义一致。
:::

### 多环境路由

如果拥有多个开发环境，可以使用环境名称指定连接目标：

```bash
# 连接默认环境
ssh -p <端口> root@<服务器地址>

# 连接指定环境（SSH 用户名为环境名称）
ssh -p <端口> <环境名>@<服务器地址>
```

::: info 路由机制
平台通过 SSH 公钥识别您的身份，因此只能连接到您自己的环境。SSH 登录名仅用于指定要连接的环境名称，不会影响用户身份验证。
:::

## 数据持久化

开发环境采用 **全量持久化** 方案：以一块 Longhorn 持久卷（即上文的工作盘 / 数据盘）作为 OverlayFS 的可写层，让容器根文件系统的所有变更都落到工作盘上，重启后完整保留。

### 工作原理

```
镜像根文件系统（lowerdir，只读）
        ↕ OverlayFS ↕
工作盘 = Longhorn PVC（upperdir/workdir，可写，挂在 /mnt/persistent）
        ↓ chroot ↓
SSH / code-server 运行在 overlay 内，其 / 即工作盘支撑的 overlay
```

- 工作盘（Longhorn RWO PVC）挂载在容器的 `/mnt/persistent`，其下的 `upper/`、`work/` 作为 OverlayFS 的可写层。
- 入口脚本把镜像根作为 lowerdir、工作盘作为 upper/work，挂出一份 overlay，并 **chroot** 进入它再启动 sshd / code-server。
- 因此通过 SSH 看到的 `/` 就是这块 overlay——往 `/` 写的任何内容（pip/apt 安装、系统配置、家目录文件等）实际都写到了工作盘上，重启不丢。

### 持久化内容

| 持久的内容 | 是否持久化 | 说明 |
|----------|:----------:|------|
| pip/apt 安装的包 | ✅ 是 | 写入工作盘（overlay 层） |
| 修改的系统配置 | ✅ 是 | 写入工作盘（overlay 层） |
| /root 家目录、代码 | ✅ 是 | 写入工作盘 |
| conda 环境 | ✅ 是 | 写入工作盘 |
| /models 中的文件 | ✅ 是 | 共享存储持久化（与工作盘独立） |
| 环境变量 | ❌ 否 | 每次启动重新生成 |

### 存储空间与扩容

- 工作盘容量在创建环境时自选（默认 200 GiB），计入[数据盘 2 TiB 配额](/guide/data-volumes#创建数据盘)。
- 工作盘支持**在线扩容**：在环境详情中扩容存储，Longhorn 会在线放大 PVC，无需重启。
- `/models` 由共享存储提供，与工作盘相互独立、互不影响。

### 删除环境与数据保留

- 删除环境时，平台只**解绑**工作盘（数据盘），**不会删除工作盘**，数据完整保留。
- 该工作盘随后可在「数据盘」页面查看，或挂到下一个环境继续使用——杜绝了删除环境后遗留的孤儿卷。
- 若确定不再需要某块工作盘的数据，请到「数据盘」页面手动删除（需先删除对应环境以解绑）。

::: warning kubectl exec 与 SSH 看到的挂载不一样
业务进程运行在 chroot 内的 overlay 中，`kubectl exec` 进入的是 chroot **之外**的容器原始 rootfs，看到的 `/`、`df -h` 与 SSH 会话不同（详见 [故障排查](/admin/troubleshooting)）。**判断持久化是否生效请以 SSH 视图为准。**
:::

## 环境状态
| 状态 | 说明 |
|------|------|
| **Creating** | 创建中 |
| **Starting** | 启动中 |
| **Running** | 运行中 |
| **Stopping** | 停止中 |
| **Stopped** | 已停止 |
| **Error** | 错误 |

除运行状态外，环境还会带一个**存储健康标记**：

| 存储健康 | 说明 |
|------|------|
| （正常） | 工作盘挂载正常 |
| **存储降级（StorageDegraded）** | 工作盘/卷 attach 失败等导致环境卡住。环境详情会出现告警条与「强制重启」按钮，详见 [故障排查](/admin/troubleshooting) |

## 使用环境

### 启动环境

1. 找到已停止的环境
2. 点击 **启动** 按钮
3. 等待状态变为 Running

### SSH 连接

1. 确认环境状态为 Running
2. 在环境详情中复制 SSH 连接命令
3. 在本地终端执行连接
4. 或使用 VS Code Remote-SSH / PyCharm 等工具连接

### 环境操作

| 操作 | 说明 |
|------|------|
| 启动 | 启动已停止的环境 |
| 停止 | 停止运行中的环境 |
| 删除 | 删除环境（仅解绑工作盘，工作盘数据保留，可在「数据盘」页面复用） |
| 扩容存储 | 在线扩容工作盘（Longhorn 支持在线扩容，无需重启） |
| 强制重启 | 工作盘 attach 失败（存储降级）时使用：先停止再启动，重新挂载工作盘 |

::: info 关于自定义镜像
开发环境使用统一的通用模板。如需使用自定义镜像进行训练或推理，请使用 [训练任务](/guide/training-jobs) 功能。
:::


## 调试节点

平台提供调试节点供用户通过 SSH 登录进行轻量级开发和调试工作。

::: danger 严禁在调试节点执行计算任务
**通过 SSH 登录调试节点（裸机）后，仅供代码编辑、文件管理和环境调试使用，严禁直接在裸机上执行计算任务**，包括但不限于：
- 模型训练、微调
- 大规模数据处理
- 长时间运行的脚本
- GPU/NPU 密集型任务

::: tip 平台内无此限制

:::
:::

## 故障排查

### 环境无法启动

1. 检查资源配置是否在配额内
2. 检查镜像是否有效
3. 查看环境事件

### SSH 连接失败

1. **确认已上传 SSH 公钥**：在 [设置 > SSH 公钥](/guide/settings#ssh-公钥管理) 中检查是否已添加公钥
2. **确认环境正在运行**：SSH 只能连接 Running 状态的环境
3. **检查密钥匹配**：确保使用的私钥与上传的公钥配对
4. **检查 Host Key 变更**：如果连接提示 host key 变更，执行：
   ```bash
   ssh-keygen -R "[<服务器地址>]:<端口>"
   ```
5. 检查网络连接是否正常

### scp / sftp 失败：subsystem request failed

新版客户端（OpenSSH ≥ 9.0，含 Windows 自带的 `scp`）默认走 SFTP 协议传输，需要服务端开启 `Subsystem sftp`。出现 `subsystem request failed on channel 0` 通常说明目标环境还运行在**旧镜像**（未开启 SFTP 子系统，或未安装 `rsync`）。

- **scp 临时救急**：加 `-O` 强制走老的 SCP 协议，不依赖子系统，新旧镜像都能用：

```bash
scp -O -P <端口> 本地文件.txt <环境名>@<服务器地址>:/models/
```

- **彻底修复**：在平台上对该环境执行**停止 → 启动**（重建 Pod 并重新拉取镜像）。若重启后依旧报错，说明节点缓存了同 tag 的旧镜像，请为镜像更换新 tag 后重新启动，或联系管理员强制重新拉取。

::: warning sftp / rsync 无法用 -O 绕过
`-O` 仅对 `scp` 有效。`sftp` 始终需要 `Subsystem sftp`，`rsync` 需要远端已安装 `rsync` 二进制——两者都必须使用已修复的新镜像。
:::

### 环境一直处于 Creating 状态 / 存储降级

1. 新环境首次启动需等待工作盘 PVC 绑定与 overlay 就绪（通常 1-2 分钟）。
2. 若环境长时间卡在 Creating，且详情页出现**存储降级（StorageDegraded）**告警，多为工作盘 attach 失败（节点重启或 Longhorn 异常）。可点击告警条上的**强制重启**（停止后重新启动，重新挂载工作盘）；仍失败可在「数据盘」页面解绑/更换工作盘后重建。
3. 若超过 10 分钟仍未就绪且无存储降级标记，检查节点资源是否充足、查看环境事件或联系管理员。

### conda 命令不可用

conda 环境在 SSH 登录后会自动加载。如果 conda 命令不可用：

```bash
# 手动加载 conda
source /root/.bashrc

# 或重新初始化
eval "$(/root/miniconda3/bin/conda shell.bash hook)"
```

### 数据相关问题

::: warning 数据安全
- **全量持久化**：容器内所有修改（pip/apt 安装、系统配置等）写入工作盘，环境重启后均会保留
- **删除环境**：仅解绑工作盘，**工作盘数据保留**（可在「数据盘」页面复用），不会自动清理数据
- **共享存储**：`/models` 路径下的数据存储在共享存储中，不受环境删除影响
- **彻底清除工作盘数据**：需先删除对应环境（解绑），再到「数据盘」页面手动删除该数据盘
- **删除用户**：将清除该用户的所有数据，包括工作盘和共享存储
:::

## 最佳实践

### 1. 合理管理环境
- 每个用户最多 2 个环境，合理规划用途
- 不使用时及时停止环境以释放资源
- 利用自动停止功能避免资源浪费

### 2. 数据管理
- **项目文件优先存放在 `/mnt/model` 目录**，该目录为大容量共享存储，适合存放数据集、模型和项目代码
- `/models`（用户主目录）存放个人配置和代码
- 工作盘（即 SSH 视角下 `/` 对应的 overlay 层）适合安装工具包，但**不建议存放大文件**——大文件会占用工作盘的 2 TiB 共享配额，且无法在多环境间共享
- 使用版本控制（Git）管理代码

### 3. 安全实践
- 使用强密钥保护 SSH 私钥（推荐 Ed25519）
- 不与他人共享 SSH 私钥
- 定期更换 SSH 密钥

## 命令行工具

开发环境预装了 `ktp` 命令行工具，可在 SSH 连接后直接使用。通过命令行提交和管理训练任务、查看日志、监控状态等。

```bash
# 查看帮助
ktp --help

# 提交训练任务
ktp submit -f job.yaml

# 查看任务列表
ktp list
```

认证 token 在环境启动时自动注入，无需手动配置。详见 [命令行工具 (ktp)](/guide/cli)。
